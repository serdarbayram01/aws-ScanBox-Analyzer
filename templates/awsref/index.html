AWS Region Latency Matrix

What it measures. Real-time TCP-connect RTT from this host (the ScanBox server you're connected to) to every public AWS region's S3 endpoint. 5 samples per target, run in parallel.

Path traversed. This host → host's Internet egress → AWS Edge / public Internet → target region's S3 endpoint. This is the same path any workload running on this host would use to call regional AWS services.

Numbers reflect. The latency your real application observes — including egress hop, Internet routing, and TCP handshake. P50/P95/P99/stdev are computed per target.

Use when. Sizing a real latency budget, picking the closest region from this host, validating that workload network performance matches expectations.

Caveat. One row of the matrix only — from this single host. For cross-host paths, deploy agents on the other side (P2P pattern).

What it measures. CloudPing.co's region × region matrix — the median latency between AWS Lambda agents running inside two AWS regions, aggregated over the timeframe you pick (P50 / 1 Day default).

Path traversed. AWS internal backbone, region-to-region private network. Not the public Internet path; no edge POP, no customer-facing routing.

Numbers reflect. AWS infrastructure baseline between regions — useful for capacity planning and comparing AWS backbone performance. Not what your real LZ workload observes (LZ egress + Internet adds ~25–35 ms on top).

Local Zone caveat. CloudPing does not measure Local Zones. We augment the matrix by mirroring each LZ's parent region row & column. Real LZ-to-LZ measurement requires an agent inside each LZ.

Source & caching. cloudping.co (community-maintained, MIT-licensed). Server caches upstream for 24 hours per (percentile, timeframe). Set the CLOUDPING_PROXY_URL env var to a benchsuite-compatible proxy to unlock all P10..P99 × 1D/1W/1M/1Y combinations.

Use when. Multi-region deployment planning, choosing a primary/secondary region pair, comparing your local probes against community-aggregated AWS backbone numbers.

What each value in the Percentile dropdown means. Pick a metric below.

P50. Half of all samples are at or below this value — the median. "If I make one request, this is the time I should expect roughly half the time." The default reference number.

Example. P50 = 18 ms means a typical round-trip is 18 ms; many requests are faster, many are slower, but 18 ms is the centre of the bell.

Read it as. "This is the typical experience."

min. The fastest single sample observed in the window. Useful as a floor — your path will never be meaningfully faster than this.

Example. min = 8 ms, P50 = 22 ms → 14 ms of overhead routinely creeps in (queueing, retries, retransmits).

Read it as. "This is the best the path can do."

P10. 10% of samples are at or below this value — the fast 10%. Tells you the lower edge of a "good run".

Example. P10 stable but P50 rising → the path *can* still hit fast numbers; something is pulling the median up intermittently.

Read it as. "This is what good days look like."

P25. The lower quartile — 25% of samples are at or below. Better than typical without being a best-case outlier.

Example. P25 ≈ P50 means the distribution is tight on the fast side; a wide gap means most requests are slower than the lucky ones.

Read it as. "This is the fast edge of normal."

P75. 75% of samples are at or below this value — the upper quartile. The boundary where "still typical" ends and "occasional slowness" begins. Useful as an early-warning signal.

Example. P50 stays flat but P75 climbs — the slow side of typical is degrading even though the median looks fine.

Read it as. "This is the slow edge of normal."

P90. 90% of samples are at or below this value — the start of the tail. The slowest 10% live beyond here.

Example. P90 = 60 ms while P50 = 25 ms → 1 in 10 requests takes 2.4× the typical time.

Read it as. "This is when things start being noticeably slow."

P95. 95% of samples are at or below this value. Captures most of the tail without being overly pessimistic. Standard SLA target for user-facing systems.

Example. "P95 latency < 100 ms" is a common SLO — meets users' patience threshold most of the time.

Read it as. "Almost everyone gets at least this good."

P98. 98% of samples are at or below this value — a strict tail. Only the worst 2% are slower. Used when even occasional slowness is unacceptable (interactive UI, real-time trading, voice).

Example. P98 spike with stable P50 → outliers got way worse while the median is unchanged; tail amplification.

Read it as. "Even the slow users mostly do this well."

P99. The worst 1% threshold. Always slower than P50; the gap between P50 and P99 measures tail latency variance — high gap means an unstable path, low gap means a consistent path.

Example. P50 = 20 ms, P99 = 30 ms → tight (good). P50 = 20 ms, P99 = 250 ms → spiky (investigate).

Read it as. "This is what your unluckiest users experience."

max. The worst single sample observed in the window. Always higher than P99. Includes brief, dramatic spikes (TCP retransmit, transient route flap, GC pause on the agent).

Example. max wildly above P99 → one-off outlier, probably not repeatable; max close to P99 → tail is genuinely heavy.

Read it as. "This is the worst it has ever been."

stdev. Standard deviation of all samples — a single number for how "wide" the distribution is. Low stdev means a tight, predictable path; high stdev means latency varies sample to sample.

Example. Two paths both have P50 = 25 ms. Path A: stdev = 3, Path B: stdev = 40 → A is steady; B is jittery and bad for latency-sensitive workloads.

Read it as. "How predictable is this latency?"
